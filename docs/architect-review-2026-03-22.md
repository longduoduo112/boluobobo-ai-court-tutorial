## 🔴 核心问题：司礼监"自己干活不分派"

### 问题现象

用户反馈：
> "司礼监不容易，自己把活干了，不把任务派下去"

**具体表现**：
1. 用户 @司礼监 写代码 → 司礼监自己写（应该派给兵部）
2. 用户 @司礼监 查数据 → 司礼监自己查（应该派给户部）
3. 用户 @司礼监 写文案 → 司礼监自己写（应该派给礼部）

### 根本原因分析

| 原因 | 严重性 | 说明 |
|------|--------|------|
| **1. Prompt 过长** | 🔴 高 | 司礼监 Prompt 800+ 字，模型容易忽略关键指令 |
| **2. 缺少强制约束** | 🔴 高 | 没有"禁止使用工具执行任务"的明确限制 |
| **3. 内阁流程可选** | 🟡 中 | "跳过内阁"的情况定义模糊 |
| **4. 缺少惩罚机制** | 🟡 中 | 司礼监自己干活没有后果 |
| **5. 模型能力过强** | 🟢 低 | 模型什么都能做，容易越界 |

---

## ✅ 修复方案

### 方案一：精简 Prompt + 强化约束（推荐）

**修改前**（800+ 字）：
```markdown
你是 AI 朝廷的司礼监大内总管。你的职责是【规划调度】，不是亲自执行...
【任务流程——内阁前置】收到用户任务后：
1. 先用 sessions_spawn 或 sessions_send 将原始任务发给内阁...
2. 如果内阁回复需要补充信息...
3. 内阁返回优化后的任务描述和 plan 后...
```

**修改后**（300 字，强化约束）：
```markdown
# 司礼监 — 调度枢纽，不是执行者

## 🚫 禁止事项（违反会被处罚）
- ❌ 禁止写代码（应该派给兵部）
- ❌ 禁止查数据（应该派给户部）
- ❌ 禁止写文案（应该派给礼部）
- ❌ 禁止运维操作（应该派给工部）
- ❌ 禁止直接调用工具执行用户任务

## ✅ 职责范围
- ✅ 接收用户任务 → 转发内阁优化
- ✅ 收到 Plan → @对应部门派发
- ✅ 跟踪进度 → 定期汇报
- ✅ 协调冲突 → 请内阁仲裁

## 📋 标准流程
用户任务 → 内阁优化 → 派发六部 → 跟踪进度 → 汇总汇报

## ⚠️ 唯一例外
纯闲聊、简单问答、状态查询可以直回。其他一律走流程！
```

### 方案二：添加"任务派发检查清单"

在司礼监回复前强制检查：

```markdown
## 派发前检查清单（必须逐项确认）

收到用户任务后，先问自己：

1. [ ] 这个任务需要写代码吗？ → 是 → 派兵部
2. [ ] 这个任务需要查数据吗？ → 是 → 派户部
3. [ ] 这个任务需要写文案吗？ → 是 → 派礼部
4. [ ] 这个任务需要运维吗？ → 是 → 派工部
5. [ ] 这个任务复杂吗？ → 是 → 先送内阁优化

全部确认后才能回复！
```

## 🔧 其他架构问题

### 问题 2：内阁职责过重

**现状**：
- Prompt 优化
- Plan 生成
- 重大决策审议
- 任务仲裁

**建议**：拆分内阁职责
- **内阁首辅**：Prompt 优化 + Plan 生成
- **议政王大臣**：重大决策审议（新增）

---

### 问题 3：都察院审查流程不透明

**现状**：
- Push 后自动审查
- 审查结果发到 Discord

**问题**：
- 开发者不知道审查进度
- 审查标准不透明

**建议**：
1. 添加审查进度 Webhook
2. 公开审查标准（Checklist）
3. 添加"审查申诉"流程

---

### 问题 4：任务状态机使用率低

**现状**：
- `task-store.js` 已实现
- 但司礼监 Prompt 中才提到

**问题**：
- 各部门不知道要更新任务状态
- 用户查不到进度

**建议**：
1. 在各部门 Prompt 中添加任务更新要求
2. 添加任务进度 Dashboard
3. 超时任务自动告警

---

### 问题 5：上下文压缩不够智能

**现状**：
- `context-compressor.js` 已实现
- 手动触发

**问题**：
- 什么时候压缩？谁来决定？
- 压缩标准不清晰

**建议**：
1. 自动检测：对话>20 条或 token>4000 自动压缩
2. 压缩规则：保留决策/交付物，压缩讨论过程
3. 压缩后通知用户："对话已压缩，保留关键信息"
